# 課堂版安全架構與上線閘門

更新日期：2026-07-14

## 目前結論

教師快派、匿名名單、全班／小組／個別目標、學生匿名加入、活動生命週期、無排名即時狀態、答案隔離、伺服器判分與全班共同 Boss 的程式垂直切片已完成。資料庫 migration 具有 RLS、最小權限 grant、`security definer` RPC、同班級目標外鍵與不可變學習事件，並由靜態契約測試保護。

這不等於已可正式上線：目前沒有為本系統建立或連結專用 Supabase 專案，migration 尚未在真實 PostgreSQL runtime 套用，受保護狀態輪詢與跨裝置流程也尚未實機驗證。前端因此在缺少公開設定時顯示安全閘門，不會模擬可用的活動碼。

## 信任邊界

| 區域 | 可以知道 | 不可以知道或直接改寫 |
|---|---|---|
| 學生瀏覽器 | 公開題幹、選項、自己的參與狀態、作答後回饋、全班共同修復值 | 作答前正解、其他學生作答、教師資料、完整名單、任意學習結果 |
| 教師瀏覽器 | 自己的班級、匿名別名／學習代碼／小組、可派的雙人複核能力、參與者支援狀態、活動結束後的匿名題目彙總 | 題庫私有答案欄、其他教師班級、個別學生選項明細、直接竄改活動生命週期或學習事件 |
| `public` schema | 班級活動投影、參與狀態、答案隔離後的活動題目、共同故事進度 | 未複核草稿的私有題庫內容 |
| `private` schema | 具版本的完整題目、正解、解析、提示、複核資訊 | 不開放給瀏覽器角色直接讀取 |

瀏覽器只使用專用專案的 publishable key。service role key、資料庫密碼及其他管理權限不得出現在 `NEXT_PUBLIC_*` 環境變數或前端 bundle。

## 身分與資料流程

1. 教師以電子郵件 Magic Link 登入；`teacher_profiles` 必須是 `approved`，且班級需明確指派給該教師。
2. 教師以匿名別名、六碼安全字元學習代碼與選填小組建立名單；介面明確禁止輸入真實姓名、電子郵件或生日。
3. 教師只能從正式發布、具有至少兩筆複核紀錄的題目建立 3／5 題活動。小組至少 2 個目標、個別恰好 1 個目標，且目標必須屬於同一個未封存班級。
4. 學生以 Supabase 匿名身分、六碼、暱稱與選填學習代碼加入。小組／個別任務會核對活動目標；全班任務可不輸入學習代碼。
5. 教師透過專用 RPC 啟動、停止加入或結束活動；瀏覽器沒有直接更新活動狀態的權限。
6. 學生取得的題目投影不含 `correct_option_id`、解析或提示答案。
7. 學生選答案後，瀏覽器先把公開題目快照、選項、匿名活動識別與裝置事件 ID 寫入 IndexedDB；本機紀錄不含正解、解析或提示答案。
8. 作答交由 `submit_classroom_response` 在伺服器查正解、判定學習結果、寫入作答與不可變學習事件，再回傳該題解析。收到伺服器判定後才移除本機待送紀錄並顯示回饋。
9. 同一位參與者、活動、題目與版本具有唯一約束；同一活動中的學習代碼也不能被兩個匿名身分占用。斷線、頁面重整或回應遺失時沿用同一裝置事件 ID 重送，避免重複貢獻。
10. 教師看板每 5 秒透過 `list_activity_participant_status` 保護性 RPC 更新支援狀態，不訂閱或直接讀取原始參與者資料；看板不顯示分數、速度、名次或完整錯題內容。
11. 課後報告 RPC 只回傳活動、參與人數、完整作答人數，以及各題獨立答對／需要支援的計數；不回傳暱稱、匿名身分 ID、選項明細、題目正解或解析。

學習代碼只是課堂活動的匿名入場憑證，不是永久帳號，也不能單獨證明學生真實身分。

## 主要防護

- 所有公開課堂資料表啟用並強制 RLS。
- 新資料表不依賴自動曝光；migration 使用明確 revoke／grant。
- 教師查詢皆同時檢查 `auth.uid()`、核准狀態與班級所有權。
- `activity_targets` 以活動、班級與匿名成員的複合外鍵限制跨班誤派；瀏覽器沒有直接新增目標的權限。
- 小組／個別加入會在 `security definer` RPC 內核對目標名單，錯誤訊息不回傳完整名單。
- 加入 RPC 對單一匿名身分設置每 10 分鐘 12 次嘗試上限；活動碼、學習代碼錯誤、代碼已被占用與逾時都回傳一致失敗結果，降低代碼探測風險。
- 學生只能讀取或提交與自己匿名身分綁定的活動資料。
- 教師瀏覽器沒有 `activity_responses`、`classroom_learning_events` 或全班 `activity_participants` 原始列的直接讀取權；即時支援與課後報告只走最小化 RPC 投影。
- 待送答案依活動與匿名參與者分區，只在目前瀏覽器保存；選項送達前鎖定，不能藉由離線模式取得作答後解析。
- 課後報告由擁有該活動且已核准的教師 RPC 產生，只提供匿名彙總；前端另設保守證據門檻，資料不足時禁止標示共通弱點。
- 活動建立、加入、啟動與作答都走具固定 `search_path` 的 RPC。
- 正解放在 `private.question_versions`，活動題目只保存題目 ID、版本與位置。
- 學習事件由資料庫 trigger 阻擋 update／delete。
- 正式題庫不自動 seed，避免把 60 題草稿誤當雙人複核正式題。

## 正式上線前硬性閘門

- 建立經使用者確認方案與費用的專用 Supabase 專案；不得借用既有無關專案。
- 在雲端設定開啟匿名登入與教師電子郵件登入，完成 redirect URL 與郵件寄送測試。
- 先執行 migration dry run，再套用並用真實 PostgreSQL 驗證所有 RLS、RPC、trigger、constraint、加入限流與狀態輪詢。
- 補管理者 bootstrap 流程，建立核准教師與班級；管理入口不得暴露 service role key。
- 在真實 PostgreSQL runtime 驗證標準化複核紀錄的唯一約束、作者不得自審、兩位不同有效複核者，以及發布前重新檢查資格的完整行為。
- 完成至少 200 題原創變式題的兩位英語教師複核、版本追蹤、授權來源與抽樣稽核。
- 完成雲端／邊緣層速率限制、跨匿名身分濫用測試、匿名帳號清理與資料保存／刪除政策；資料庫內的單一匿名身分限流不能取代網路層防護。
- 進行真實桌機、手機、平板跨裝置活動，以及鍵盤、螢幕閱讀器與弱網路測試；目前 IndexedDB 保存、自動重送與事件冪等只有自動化測試證據。
- 以真實班級活動驗證課後彙總 SQL、缺席／斷線資料區分、門檻敏感度與補救建議可用性；目前只有純函式、RPC 契約及元件測試。
- 由遊戲設計、英語教學、兒童 UX 三種角色獨立做 Octalysis 中位數驗收；即使達 70／80，兒童安全閘門未過仍不得發布。
